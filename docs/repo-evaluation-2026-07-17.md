# JH Prototype Playground — Full Repository Evaluation

**Date:** 2026-07-17
**Reviewer:** Principal-level engineering / design-systems / security pass (Claude Code, with four parallel sub-agents + empirical browser and build verification)
**Commit reviewed:** `966d69a` (working tree had one uncommitted change: `M src/data/features.json`)
**Method:** `npm install` → `tsc -noEmit` → `vite build` → `vite build` with `VITE_EXTERNAL_BUILD=true` on a pruned copy → docs `--check` → live dev server driven in Chrome (gallery + 3 prototypes + Features + Resources, light & dark). Every finding is tagged **VERIFIED** (I read the code / reproduced it) or **SUSPECTED** (reasoned but not executed).

## Executive summary

**Overall health: good for its purpose, with a small number of real risks.** This is a well-crafted, thoughtfully-commented internal tool. TypeScript is strict and compiles cleanly (`tsc -noEmit` exit 0), the production build succeeds, the generated docs are in sync (`check-docs` exit 0), prototype auto-discovery uses a smart eager-meta / lazy-code split, listener cleanup is disciplined, and the dev endpoints already use path-traversal guards and per-file write locks. The app renders correctly and without console errors in both themes, and text contrast passes WCAG AA in light and dark.

The problems cluster in three places: (1) the dev-only `/__proto-api` endpoints are reachable cross-origin from a malicious webpage because the custom middleware is registered *before* Vite's CORS/host-check, and one of them corrupts source files via a `String.replace` pattern bug that chains into same-origin code execution; (2) the external, credit-union-facing build leaks the internal Features board (14 cards with designer names) into the shipped bundle; and (3) the repo's own token discipline is widely violated — including **two token families documented in `CLAUDE.md` that don't exist in the vendored tokens at all**, which means the standard the playground enforces on designers is itself partly wrong.

Note: the evaluation brief referenced a `run-agent` endpoint "that spawns a headless AI CLI and streams output." **No such endpoint exists in this repo** (verified by grepping all of `src/`, `scripts/`, `vite.config.ts` for `spawn`/`exec`/`run-agent`). The only process spawning is fixed-argument `git`/`npm` calls with no shell and no attacker-controlled argv. That whole class of concern is therefore moot here.

### Top 5 risks
1. **CSRF file-write on every `/__proto-api` POST** — a malicious site a designer visits while running `npm run dev` can silently rewrite any `meta.ts`, flip `public: true` on a private prototype, or overwrite `.designer.local.json`. Vite's CORS/host-check never runs for these routes. (Security S1 — High)
2. **`String.replace` special-pattern injection in the meta writer** — a title/description containing `$'`, `` $` ``, `$&`, or `$1` corrupts `meta.ts` on write, and because every `meta.ts` is eagerly imported, a crafted value becomes same-origin code execution. Also a plain data-loss bug a designer can trigger by accident. (Security S2 / Arch C5 — High)
3. **Internal Features board leaks into the external CU-facing bundle** — `prune-private-prototypes.ts` blanks `external-links.json` but not `features.json`, and `proto-features` is imported unconditionally. All 14 feature cards (with `submittedBy`/`assignedTo` names) ship in the public bundle. (Security S4 — High, verified by building it)
4. **`CLAUDE.md`'s token table documents nonexistent tokens** — `--jh-font-weight-regular/-semibold/-bold` and `--jh-color-content-warning-enabled` do not exist in `@jack-henry/jh-tokens`; ~44 references across the repo silently ride on fallbacks, and one (`container-surface-enabled` in member-overview) falls back to a hardcoded light-blue that breaks in the default dark theme. (Docs H1/H2, Compliance A — High)
5. **1.04 MB entry chunk** — every route's component graph (highlight.js, the full component registry incl. `jha-advanced-table`, kanban board, source viewer) is statically imported, so even a visitor who only opens the gallery downloads all of it. Matters most for the GitHub Pages external gallery handed to stakeholders. (Arch A1 — Medium)

### Top 5 quick wins
1. **Blank `features.json` in the external prune step** (one line mirroring the existing `external-links.json` handling) — closes the external leak. Effort S.
2. **Fix `upsertMetaField` to use a replacement *function*** (`(_, p1) => p1 + serialized`) — kills both the data-corruption bug and the code-exec escalation. Effort S.
3. **Add an `Origin`/`Sec-Fetch-Site` allowlist check** at the top of each `/__proto-api` handler — closes the CSRF class. Effort S.
4. **Correct `CLAUDE.md`'s token table** (named weights → `--jh-font-weight-400/500/700`; delete the `warning` color row or point to `--jh-color-yellow-*`) and fix `_template/index.ts`, which seeds the `font-size-500`/`font-weight-bold`/rem-fallback mistakes into every new prototype. Effort S.
5. **Fix the two broken `jh-select`s** in `annotated-transfer-flow` (slotted `<jh-list-item>` → `.options`) and the raw `<button>` in the `login-flow` reference prototype. Effort S.

## Per-dimension detail

### 1. Architecture & structure
The shape is sound and mostly scales. **Prototype discovery is the highlight**: `meta.ts` globs are eager (cheap, needed for the gallery), `index.ts` globs are lazy (`proto-shell.ts:207`), and `?raw` source is a separate lazy glob — so adding prototypes grows only the meta payload, and each prototype's component graph loads on demand (VERIFIED). State ownership is clean: the designer profile is a well-designed module singleton with a single readiness promise and file-backed persistence (`utils/designer-profile.ts:69`), and Features/external-links use optimistic updates with rollback and server-side vote increments to avoid clobbering.

The structural weaknesses are (a) **everything except prototypes rides in the 1 MB entry chunk** (A1), and (b) **copy-paste convergence** — the `/__proto-api` fetch wrapper (8×, C1), the dialog overlay (3×, C2), the anchored-menu math (2×, C3), and discovery logic (3×, A4) each exist in multiple places. For a repo whose primary contributors are AI assistants pattern-matching on existing code, this duplication is the thing most likely to drift. The hash router works (back/forward verified) but has no explicit not-found handling and scatters route strings as literals (A3). Dead code: the `ProtoCard` element is never rendered even though the whole app's meta types live inside it (A5).

### 2. Adherence to the repo's own rules
This is where the volume is. Across 41 UI files the sub-agent found **~84 references to tokens that don't exist**, 12 `font-size` intent mismatches (gotcha 3d — e.g. `var(--jh-font-size-500, 1.5rem)` renders 20px, not the intended 24px), ~132 misleading rem dimension fallbacks (gotcha 3c — inert, but propagated by `_template`), the two broken `jh-select`s (CMP-B), and a raw `<button>` in the reference prototype (CMP-C). Most consequential:
- The nonexistent tokens trace back to **`CLAUDE.md` itself being wrong** (D1) — this isn't designers going rogue, it's the spec.
- `member-overview` uses `--jh-color-container-surface-enabled` with light-blue hex fallbacks that **visibly break in the default dark theme** (CMP-D) — the one place a phantom token isn't harmless.
- `_template/index.ts` seeds `font-size-500`, `font-weight-bold`, and rem fallbacks into every future prototype (CMP-E).

Zero violations of gotcha 3b (line-height token misuse) anywhere, and every prototype correctly uses `export default class`, no `@customElement` on the default export, no `min-height:100vh`. Fully clean: `warning-management-v1/index.ts`+`routing.ts`+`store.ts`, `blank-canvas`, all 10 `meta.ts` files, `index.html`.

### 3. TypeScript & correctness
`tsc -noEmit` exits **0** (VERIFIED — no errors, no warnings). tsconfig is strict with `noUnusedLocals/Parameters`. Gaps: no `noUncheckedIndexedAccess`/`noImplicitOverride`/`noFallthroughCasesInSwitch` (T6); 25 unsafe `as HTMLInputElement` casts on web components (T2); a stale-async load race (T1); unvalidated JSON/localStorage casts (T4); one floating clipboard promise (T5); a theoretical tag-name collision (T7). None block compilation; T1 and S2/C5 are the ones with real failure modes.

### 4. Accessibility (verified in-browser, light + dark)
Drove the app in Chrome. **Positives:** gallery rows are keyboard-focusable and Enter navigates (VERIFIED — pressing Enter on a focused row routed to the prototype); the advanced table exposes proper sort buttons, column headers, and select-all/row checkboxes in the a11y tree; theme toggle has a correct `aria-label`; no console errors on any page. **Contrast passes AA in both themes** (VERIFIED by sampling computed tokens): light secondary text `#636363` on `#ebebeb` ≈ 4.97:1; dark secondary `#afafaf` on `#202020` ≈ 7.3:1. **Gaps:** gallery rows render as `generic` (no `list`/`listitem` or `button` role) — a `jh-list-item` limitation, keyboard-operable but semantically thin; several prototypes use clickable `<div tabindex="0">` / emoji-as-icon (CMP-G) which are custom controls without full button semantics. No focus-trap issues observed in the dialogs during the pass, but a dedicated keyboard-only sweep of each dialog wasn't exhaustive (SUSPECTED-clean, not proven).

### 5. Security & safety
Judged against "dev-only, localhost." The important nuance: **a malicious website open in the designer's browser can issue simple cross-origin POSTs to `http://localhost:5173/__proto-api/*`**, and because the custom middleware runs before Vite's CORS/host-check, those writes succeed (S1). Chained with the meta-writer `replace` bug (S2), that escalates to same-origin code execution via eagerly-imported `meta.ts`. Separately, the external build leaks the Features board (S4, VERIFIED by building it). **Checked and clean:** path traversal (meta writer enforces `^[a-z0-9-]+$` + `path.resolve` + `startsWith` guard), command/argument injection (all subprocesses use `execFileSync`/`spawnSync`, fixed argv, no shell), write races (`withFileLock`), committed secrets (none — `.npmrc` uses env expansion, MCP config holds only public URLs, all sensitive files gitignored), and GitHub Actions pwn-request patterns (no `pull_request_target`, no untrusted-ref checkout-and-execute, no `${{ }}` script injection). One context amplifier: per project memory both repos are currently **public**, which makes the `designer`/`name` slugs needed for S1/S2 payloads trivially discoverable — worth resolving the repo-visibility question. Confirm `secrets.EXTERNAL_REPO_TOKEN` in `deploy-external.yml` is fine-grained/least-privilege (can't be verified from repo contents).

### 6. Performance
Build succeeds in ~1.3 s. The dominant cost is the **1.04 MB / 222 KB-gz entry chunk** (A1) — Rollup itself warns about the >500 KB chunk. Prototypes lazy-load correctly, so this is fixed-overhead paid by every visitor including the external gallery. Secondary: source-drawer re-highlighting on unrelated updates (P2), synchronous `git fetch` in a middleware (P3b), per-render array recreation causing header re-renders (P4), and 3× full list rebuilds per gallery keystroke (trivial at 7 prototypes). Update polling is well-behaved: first check at 60 s, then every 30 min, cleared on disconnect (VERIFIED). Listener cleanup is symmetric across all components (VERIFIED — a genuine strength).

### 7. Documentation accuracy & drift
Generated-file hygiene is **perfect**: `check-docs` exits 0, `.cursorrules` is byte-identical to `CLAUDE.md`, all four `.cursor/commands/*.md` match `.claude/commands/*.md`, and `.mcp.json`/`.cursor/mcp.json` match `scripts/mcp-servers.js` through the documented transform; all generated files are gitignored as claimed. The token *values* are almost all exact (every dimension, font-size, line-height, weight-*number*, and the container/divider color names verified against the vendored CSS; the `font-size-100/-200` and `content-tertiary-enabled` "these don't exist" claims are **true**). The two real doc bugs are the fabricated named font-weights and warning color (D1). Plus the template-copy import path (D-M1), 6 undocumented meta fields (D-M2), and minor README items (D-L). A large set of hand-written architectural claims (auto-discovery, shell wrapping + definite height, navItems, external-build gating, password flow, `.designer.local.json` pattern, jh-select `.options`) were all **verified accurate** with `path:line` evidence.

### 8. Testing & tooling
**No tests, no test runner, no lint/format tooling** (X1, X2). The npm scripts are otherwise coherent (`build` runs `tsc` first; `check-docs` enforced in CI; `postinstall` regenerates configs). The highest-value test target is the pure Node string-surgery in `vite.config.ts` (`upsertMetaField`/`extractMetaField`) — it edits designers' source files and currently has a live bug (S2). `dev:external` previews the external UI gating but doesn't run the prune, so it won't surface content-leak issues like S4 — worth a note.

### 9. Consistency & quality
Naming (`proto-*`, `_camelCase` state) and comment discipline are consistently good — comments explain *why* (e.g. the composedPath-vs-stopPropagation note in `proto-shell.ts`). The main inconsistencies are the duplicated infrastructure (C1/C2/C3/A4), the mixed event-typing patterns (T2), bare custom event names (`close`/`saved`/`cancel`) that could collide, and inconsistent form-state prefixes (`_fUrl` vs `_draftTitle`).

## Suggested new features / to-do items
Framed as additions to the Features board:
1. **CI lint gate for token/rule compliance** — a small script (extend `audit-component-docs.ts`) that fails the build on nonexistent-token references and raw HTML controls in `src/prototypes`. Turns the rules the AI is *asked* to follow into ones that are *enforced*. (This directly addresses the largest finding cluster.)
2. **External-bundle leak test in CI** — after `build:external`, grep the output for known-internal markers (feature titles, designer emails) and fail if present. Would have caught S4.
3. **A "New prototype" scaffold that writes a correct, token-accurate `meta.ts`/`index.ts`** so designers stop copying `_template`'s mistakes (complements the CMP-E/`/new-prototype` fixes).
4. **Origin-locked dev endpoints** as a first-class setting, plus a short SECURITY.md documenting the dev-server threat model.
5. **Mobile viewport preview toggle** and **one-click prototype duplication/versioning** — both already on the board; both are reasonable and low-risk.
6. **Route-level code splitting** surfaced as a "faster external gallery" item (A1) — user-visible on the CU-facing site.
