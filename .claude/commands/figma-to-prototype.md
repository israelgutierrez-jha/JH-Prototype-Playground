# /figma-to-prototype

Scaffold a new prototype in the JH Prototype Playground **from a Figma design** instead of a text description — rebuilding it using only real JH web components.

## Prerequisites

This command needs the `figma` MCP server (pre-configured in this repo's `.mcp.json` / `.cursor/mcp.json`). If Figma tools aren't available, tell the user to authorize it first — in Claude Code, run `/mcp` and choose "Authenticate"; in Cursor, open Settings → MCP and authorize the `figma` server. This is a one-time step per machine.

The `chrome-devtools` MCP server (also pre-configured, no auth needed) lets you visually verify the finished prototype against the Figma screenshot instead of just telling the user to eyeball it — see step 10.

## Steps

1. **Ask for the Figma URL first, and only that.** One link either way — never ask the designer for multiple URLs:
   - **Single screen:** a frame URL that includes a `node-id` (e.g. `https://figma.com/design/:fileKey/:fileName?node-id=1-2`).
   - **Multi-screen flow:** ask them to select the Section (or parent frame) that contains every screen in the flow as a child frame, and paste that link instead. If they haven't grouped the screens yet, tell them how: select all the screen frames on the canvas → right-click → "Section" (or drag them into an existing Section) → name each child frame with a leading number in flow order, e.g. `1 - Login`, `2 - Enter code`, `3 - Success`. This numbering matters — see step 3.
   - If the URL has no `node-id`, ask for a node-specific link — don't guess one. Don't ask for name/tags/description in this same message; wait for the URL before moving on.

2. Extract `fileKey` and `nodeId` from the URL.

3. **Determine single screen vs. multi-screen flow before fetching anything else:** call `get_metadata(nodeId, fileKey)` on the node from the URL. If it has multiple direct child frames that look like distinct screens (not just nested groups/shapes within one screen), treat it as a flow — record each child's own `nodeId` and name. Sort them by their leading number if the frames are numbered (recommended, see step 1); if none are numbered, fall back to top-to-bottom/left-to-right canvas position from the metadata, but flag to the user that the order is a guess and confirm it with them before scaffolding. If the node has no such children, it's a single frame — continue as before.

4. Fetch the design:
   - **Single frame:** `get_design_context(nodeId, fileKey, clientFrameworks: "lit,web-components", clientLanguages: "typescript")`, `get_screenshot(nodeId, fileKey)`, and `get_variable_defs(nodeId, fileKey)` on that one node.
   - **Multi-screen flow:** repeat all three calls once per child frame's `nodeId` from step 3 — never one call on the parent Section/frame. A single call on a large parent risks Figma's own size-based fallback silently returning metadata instead of code once the section holds more than a couple of screens (`get_design_context` does this automatically when the output would be too large).
   - Leave Code Connect enabled either way (don't set `disableCodeConnect`). Today this will likely come back unresolved for most nodes since the JH Figma library has no Code Connect mappings yet — that's expected, not an error. If the design system team adds mappings later, this same call starts returning real `jh-*` code automatically, no changes needed here.

5. **Now collect the remaining details, one field at a time, in this order: designer name → prototype name → tags → description.** Don't dump all of these into one message — give each its own turn, and pick the presentation that fits the field:
   - **Designer name.** Check for an existing designer profile first, but verify it against real folders before trusting it: look for `.designer.local.json` at the repo root. If it exists and its `name` field is non-empty, kebab-case it (lowercase, non-alphanumeric → `-`, collapse/trim hyphens) and list the folders under `src/prototypes/*/`. If the derived slug exactly matches one of those folders, use it and skip asking — keep the profile's `name` as-typed (e.g. "Israel Gutierrez") for `designerName` in step 9, don't kebab-case it for that field. If it doesn't match any existing folder (typo, name changed, or a stale profile), ask once with `AskUserQuestion`: offer the existing designer folders as options plus the derived slug itself (in case it's really a new designer), then use whichever they pick — don't silently create a new folder on a mismatch. If the file is missing or empty, list the folders under `src/prototypes/*/` and ask with the `AskUserQuestion` tool: offer the existing designer folder names as options (recommend whichever seems like a returning designer) so they reuse their folder, plus the built-in "Other" for a new name. If there are no prototypes yet, or `AskUserQuestion` isn't available in the current tool (e.g. Cursor), just ask directly: "What's your name? (e.g. Israel Gutierrez — used for both the folder name and how it's displayed)", keeping their answer in natural form for `designerName`. Either way, once confirmed, write `.designer.local.json` as `{"name": "<real name, not kebab-cased>", "onboarded": true}` (create or overwrite) so future runs and the browser app pick it up without asking again. This file is gitignored and local to this machine — never commit it.
   - **Prototype name.** Ask directly as free text, e.g. "What should this prototype be called? (e.g. `account-transfer-flow`)". This has no meaningful fixed option set — don't force it into a picker.
   - **Tags.** Scan existing `meta.ts` files across `src/prototypes/` for tags already in use. Ask with `AskUserQuestion`, `multiSelect: true`, offering the most common existing tags as options plus "Other" for anything new. If `AskUserQuestion` isn't available, ask for comma-separated tags directly.
   - **Description.** Ask directly as free text, 1–2 sentences on what the prototype demonstrates. Since the Figma frame(s) and screenshot(s) are already fetched by this point (step 4), draft a one-sentence description from what's visible and offer it to the user to confirm or edit, rather than starting from a blank prompt.

6. **Ask about interaction behavior a static frame can't show.** A Figma frame only shows one state — it can't tell you *how* a screen gets there. Before mapping components or writing code, look at what you fetched in step 4 for signals worth confirming, for example:
   - A text input positioned next to/above a list, card, or results area → ask whether results should update live as the user types (each keystroke), after a debounce, or only after an explicit action (button click, Enter key). This is exactly the kind of thing a static frame can't answer — the design can't distinguish "results already loaded" from "results that appear after you start typing."
   - A form with several fields feeding one downstream action → ask whether validation happens per-field (on blur), on submit only, or live as you type.
   - A toggle/select/checkbox that plausibly filters or shows/hides other content → confirm whether it actually should, and what the unfiltered/default state looks like.
   - Any loading/pending/skeleton state implied by the design (a spinner icon, a "Scanning…" label, a progress bar) → confirm what triggers it and roughly how long to simulate it.
   - A dataset shown with only one or two example rows/cards → confirm whether that's the intended full set or a representative sample you should pad out.

   Only ask when the frame is genuinely silent on something that would materially change what you build — don't ask about behavior a plain reading of the design already makes obvious (a "Submit" button clearly submits). If nothing raises this kind of ambiguity, skip this step without asking. Prefer `AskUserQuestion` with concrete, grounded options over an open-ended question — e.g. "Live as you type" / "After a button click" / "Other" — since drafting real options requires you to have actually looked at the design, and a multiple-choice question is faster for the designer to answer than free text.

7. Map every distinct element in the layer tree to a real `jh-*` component:
   - Use the same matching `/use-jh-component` does — layer/component names, structure, and the screenshot as evidence, cross-referenced against the component reference in `CLAUDE.md` and `src/data/components/`.
   - `search_design_system` can help confirm what a given Figma component instance actually is if its layer name is ambiguous.
   - If an element's role is ambiguous, resolve it to the closest reasonable `jh-*` component — don't guess wildly.
   - **If there's truly no reasonable `jh-*` match** (e.g. a decorative illustration or one-off shape with no component equivalent), stop and ask the user for explicit approval before writing any non-JH markup. Never invent custom HTML/CSS silently — this is a hard rule from the root `CLAUDE.md`. If the closest available option is a legacy `@banno/jha-wc` component (e.g. `jha-dialog` for a modal — `jh-elements` has no dialog/modal yet), that counts as a reasonable match, not a gap to ask about; check `src/data/components/` for a documented legacy fallback before concluding there's no JH equivalent.

8. Resolve colors, spacing, and typography from `get_variable_defs` to `--jh-*` tokens:
   - Match by variable name first (e.g. a Figma variable named like a JH token), nearest value second, using the JH Design Tokens table in `CLAUDE.md`.
   - Keep the `var(--jh-token, literal-fallback)` pattern used throughout the codebase (see `src/prototypes/example/login-flow/index.ts`), with the fallback matching the raw value from Figma.

9. Scaffold the prototype exactly like `/new-prototype`:
   - Create `src/prototypes/[designer]/[prototype-name]/` (with the first-time-designer `CLAUDE.md`/`.cursor/rules` scaffolding if this is their first prototype).
   - `meta.ts` — same shape as `/new-prototype`.
   - `index.ts` — build the markup and styles from the resolved Figma structure instead of freeform generation, applying whatever interaction behavior was confirmed in step 6.
   - **For a multi-screen flow** (step 3): each child frame becomes one `@state()`-driven render branch, in the order determined in step 3 — same pattern as `login-flow`. For each screen, identify its primary call-to-action (the button that visually reads as the flow's forward action: `appearance="primary"`, or labeled Continue/Next/Submit/Confirm/Done/Save) and wire its `@click` to advance `_step` to the next screen's key. If a screen has more than one plausible forward action, or none, ask the user which button should advance the flow rather than guessing. **Before finalizing, summarize the inferred flow back to the user** (e.g. "Login → [Continue] → Enter code → [Verify] → Success") so they can correct it — this is inferred from frame naming/position and button labels, not read from Figma's own prototype-interaction wiring, which the MCP tools this command uses don't expose (see Important rules).

10. Verify visually if you can, then hand off:
   - If `npm run dev` is already running (check for a listener on its port, e.g. `lsof -i :5173`) and `chrome-devtools` MCP tools are available, navigate to the new prototype's URL, take a screenshot, and compare it against the Figma screenshot(s) from step 4. For a multi-screen flow, click through each transition in the running prototype and compare every screen against its corresponding Figma screenshot, not just the first one. Fix any obvious mismatches (spacing, missing elements, wrong component variant, a transition that doesn't fire) before handing off — don't just describe what you'd fix, actually fix it.
   - If the dev server isn't running or `chrome-devtools` isn't available, say so plainly rather than claiming visual parity you haven't checked.
   - Either way, tell the user the path to their new prototype and the `npm run dev` URL, and to iterate with `/use-jh-component` for any remaining prop-level fixes.

## Important rules

- Same hard rules as `/new-prototype`: no `@customElement`, no Tailwind/Bootstrap/external CSS libraries, no raw `<button>`/`<input>`/`<select>`, always `--jh-*` tokens.
- Never write markup for an element that has no `jh-*` equivalent without the user's explicit go-ahead first.
- Don't wait on Figma Code Connect mappings to exist — this command works today via heuristic matching, and gets more accurate for free if/when the design system team adds Code Connect for the JH library.
- **A Figma frame's width/height is just the artboard bounding box — never carry it over as a fixed CSS `height`/`min-height`/`width` on the prototype's outer container(s).** The frame size determined how the design happened to be laid out in Figma at design time; it is not a layout constraint for the web prototype, which must fill whatever space the playground's platform shell content area actually gives it. Use flexible sizing (`flex: 1`, `min-width: 0`, content-driven height) instead of copying pixel dimensions from `get_metadata`/`get_variable_defs`. A fixed width on one specific element (e.g. a side panel/nav rail that's intentionally a fixed rail width by design) is fine — a fixed height/width on the top-level layout wrapper that exists only because that's how big the Figma frame was is the anti-pattern to avoid.
- **If the frame has an element meant to reach the full height of the screen** (a side panel/nav rail with a divider that should run edge-to-edge, a split-pane layout, etc.), don't fake it with a fixed pixel height either — opt into the real available height instead. See the "Full-height split-pane layout" pattern in the root `CLAUDE.md` for the exact CSS shape and why it works.
- **Figma's own prototype interactions (the click-to-navigate wiring set up in Figma's Prototype tab) are not exposed by `get_metadata`/`get_design_context`/`get_screenshot`.** For a multi-screen flow, screen order and which button advances to the next screen are always inferred (from frame naming/position and button labels), never read from real interaction data — that's why step 1 pushes designers toward numbered frame names, and step 9 requires confirming the inferred flow with the user before finalizing instead of assuming it's right.
